404 Error handler

If a user tries to access a page that is not on the server your 404 error handler will fire and you can choose to serve up a webpage to display. The 404 error handler is simply a function and in that function you can do things (whatever you like) such as (but not limited too) serving a 404 page.

If you need a design template for a 404 page then check out some of our (pre-made) designs for 404 pages here.


500 Error handler

Internal server errors happen all the time and you need a function to catch it. Keep in mind that there are a wide range of internal server errors that can occur and there are a number of reasons why they occur (from poor code in your app, to bad code on your server, or even missing files). For example if you have a route and you accidentally deleted the webpage to be served with that route then your server will generate a 500 error.

Inside of the 500 error handler you can choose to do a number of things such as retrieve information from a database or close connections. The most common thing to do is serve up a webpage letting the user know that your server is under some sort of maintenance.

If you need a design template for a 500 page then check out some of our (pre-made) designs for 500 pages here.

UncaughtException Error Handler

One thing about Node apps is that Node has a tendecy for crashing due to several things such as loss of memory or memory leaks, or bandwith issues etc. Many of these errors are not accounted for in the code (mainly because it would be a lot of code to write) and therefore we must set whats called an
UN-CAUGHT-EXCEPPTION error handler.
This will handle all the un-caught errors.

For much of these errors the only thing that can possibly cure the problem is a restart of the app. This is where your Process Manager comes in (such as PM2 or Forever). However if your app crashes you still want to try to gracefully shutdown the Node server.



Gracefully shutting down Node is important so you can free up memory and the port on which Node was started. If your app doesn't gracefully shutdown it can hang on the port which will make you have to manually change the port number in the code. The uncaughtException Error Handler must be the last function in the script.

Keep in mind, when uncaughtException Errors happen your server is still live and able to do things (such as gracefully shutdown Node). Depending on what caused the error you should be able to manipulate Node and then shutdown.
